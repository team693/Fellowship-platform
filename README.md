# Heal Digital Fellowships Platform

A production web app for paid, verifiable digital fellowships. Students enroll
via partner-sponsored seat codes, work through immersive simulations and case
studies (hosted natively as embedded HTML modules), and earn a credible,
independently verifiable certificate on completion.

- **Framework:** Next.js (App Router) + TypeScript → Vercel
- **Backend:** Supabase (Postgres, Auth, Storage, Row-Level Security)
- **Styling:** Tailwind CSS, Heal brand system (teal / mint / coral / blue / gold)
- **Fonts:** Bricolage Grotesque, Inter, IBM Plex Sans/Mono, Noto Nastaliq Urdu (bilingual EN/UR)

---

## 1. Quick start (local)

```bash
npm install
cp .env.example .env.local        # then fill in your Supabase keys
npm run fetch:assets              # self-host D3 + fonts for embedded modules
npm run dev                       # http://localhost:3000
```

You must create a Supabase project and run the migrations first — see below.

Useful scripts:

## 2. Supabase setup (manual steps you must do)

### 2.1 Create the project
1. Go to <https://supabase.com/dashboard> → **New project**. Pick a region and a strong DB password.
2. Open **Project Settings → API** and copy:
   - **Project URL** → `NEXT_PUBLIC_SUPABASE_URL`
   - **anon public** key → `NEXT_PUBLIC_SUPABASE_ANON_KEY`
   - **service_role** key → `SUPABASE_SERVICE_ROLE_KEY` (server-only, keep secret)

### 2.2 Run the migrations
In the dashboard **SQL Editor**, run these files **in order** (copy/paste each and click Run):

1. `supabase/migrations/0001_schema.sql` — tables, helper functions, triggers
2. `supabase/migrations/0002_rls.sql` — Row-Level Security policies
3. `supabase/migrations/0003_storage.sql` — private spotlight-photos bucket + policies
4. `supabase/migrations/0005_redeem_function.sql` — atomic seat-code redemption RPC
5. *(optional)* `supabase/migrations/0004_seed.sql` — one example fellowship with the two shipped example modules

> Prefer the CLI? `supabase link` then `supabase db push` will apply everything in `supabase/migrations`.

### 2.3 Configure Auth providers
**Authentication → Providers:**
- **Email**: enable **Email**. Magic links are on by default. (No password needed.)
- **Google**: enable **Google**, paste your Google OAuth **Client ID** and **Client secret**
  (create them in Google Cloud Console → OAuth consent + Credentials).

**Authentication → URL Configuration:**
- **Site URL**: your app URL (`http://localhost:3000` for dev, your domain in prod).
- **Redirect URLs**: add both:
  - `http://localhost:3000/auth/callback`
  - `http://localhost:3000/auth/confirm`
  - (and the production equivalents once deployed)

**Magic-link email template (optional):** the default template works via the
PKCE `?code=` flow. If you prefer the token-hash flow, set the Magic Link
template link to:

```
{{ .SiteURL }}/auth/confirm?token_hash={{ .TokenHash }}&type=email
```

Our `/auth/confirm` route handles **both** `token_hash` and `code`, so either
configuration works.

### 2.4 Storage
Migration `0003_storage.sql` already creates the **private** `spotlight-photos`
bucket with a 5 MB limit and image-only MIME restriction, plus the RLS policies.
Confirm it exists under **Storage** in the dashboard. Keep it **private**.

### 2.5 Make yourself an admin
Sign in once (so your `profiles` row exists), then in the **SQL Editor**:

```sql
update public.profiles set role = 'admin' where email = 'you@healsocialfoundation.com';
```

Admins get the **Admin** nav link and can manage partners, codes, and content.

## 3. Deploying to Vercel
1. Import the repo in Vercel.
2. **Settings → Environment Variables**, add all keys from `.env.example`
   (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`,
   `SUPABASE_SERVICE_ROLE_KEY`, `NEXT_PUBLIC_SITE_URL` = your domain,
   `NEXT_PUBLIC_SPOTLIGHT_BUCKET`).
3. Add your production domain to Supabase **Redirect URLs** and **Site URL**.
4. Push to `main` — Vercel auto-deploys. `public/vendor` + `public/fonts`
   (generated by `npm run fetch:assets`) are committed, so modules stay
   CDN-independent in production too.

## 4. Embedding your own HTML modules

Drop your self-contained HTML files into **`public/simulations/`**, then create a
module in the admin UI (**Admin → Fellowships → a fellowship → Add a module**),
set its `asset_path` to the filename, and choose its **completion rule**.

Your modules render inside a **sandboxed iframe** served by `/embed/[id]`, which:
- injects `window.__HEAL_MODULE_ID__` and the `healComplete` / `healProgress` helpers,
- **rewrites CDN references** (D3 from cdnjs/jsdelivr/unpkg/d3js.org, and Google
  Fonts) to the self-hosted copies under `/vendor` and `/fonts` — so your existing
  dashboards work **unchanged** on locked-down partner networks.

### Two completion rules

**`engagement`** (explore-only dashboards): the HTML reports nothing. The parent
tracks engagement — a server-recorded dwell time (`completion_config.min_seconds`)
plus a student-clicked *Mark complete* gate — and writes progress. Your existing
dashboards work in this mode with **zero changes**.

**`reported`** (assessed modules): the HTML decides completion and posts a result.
Drop this snippet into your module and call `healComplete(score)` when the student
finishes/passes:

```js
function healComplete(score) {
  parent.postMessage(
    { type: "HEAL_MODULE_COMPLETE", moduleId: window.__HEAL_MODULE_ID__, score },
    "*"
  );
}
// call healComplete(85) when the student finishes/passes
```

### The postMessage contract

```js
// From iframe → parent:
{ type: "HEAL_MODULE_COMPLETE", moduleId: string, score?: number, meta?: object }
{ type: "HEAL_MODULE_PROGRESS", moduleId: string, percent: number }  // optional
```

The parent verifies the message came from the current module's iframe, that the
shape is valid, and that `moduleId` matches — then treats `score` as **untrusted**
input, re-validated and clamped server-side before it can count toward a
certificate.

See the two shipped examples: `public/simulations/example-explore.html`
(engagement) and `public/simulations/example-assessed.html` (reported).

## 5. How security & integrity work (important)

Certificates are only as credible as the completion data behind them, so
completion is **never trusted from the client**:

- **`progress` has no client write policy.** Under RLS, students can *read* only
  their own progress and cannot INSERT/UPDATE it at all. All progress writes go
  through server routes (`/api/progress/*`) using the service role, only after
  server-side validation.
- **Engagement completion** is validated against a server-recorded `started_at`
  timestamp — the client can't backdate time-on-module.
- **Reported scores are untrusted.** The server clamps to `[0, 100]` and enforces
  the module's `pass_score`; a failing score does not complete the module. (A
  reported module is ultimately client-authored, so for high-stakes assessment
  prefer server-authored grading; the contract already routes every reported
  result through server validation.)
- **Certificates are immutable.** A DB trigger freezes all core fields once
  issued; only `status` can change (for revocation, a service-role admin action).
  There is a `unique(user_id, fellowship_id)` backstop and issuance is idempotent.
- **Public verification is column-limited.** `/verify/[id]` reads the
  `certificate_verifications` view, which exposes only name, fellowship, issue
  date, and status — never `user_id`. UUID ids make enumeration impractical.
- **Seat-code redemption is atomic.** A single `SECURITY DEFINER` RPC locks the
  code + partner rows, marks the code, creates the enrollment, and increments the
  seat count in one transaction — no double-redemption or seat overrun.
- **Sandboxed embeds.** Modules run in an `allow-scripts` iframe **without**
  `allow-same-origin`, so they run in an opaque origin and cannot read the app's
  cookies/storage. The parent binds every message to its specific iframe
  (`event.source`).
- **Secrets** live only in env vars. The service-role key is imported through a
  `server-only` module so it can never leak into a client bundle.

### Spotlight consent (opt-in personal-data publication)
- Consent is a **deliberate, unchecked action** — never pre-selected, never
  bundled with enrollment or completion. The exact scope is stored in
  `consent_scope` with granted/withdrawn timestamps.
- Photos live in a **private** bucket. A student can upload/replace/delete only
  their own photo; **admins can read a photo only while `consent_status =
  'granted'`** — enforced by RLS on both the table and Storage.
- Withdrawal is immediate (student leaves the admin queue) and a hard
  *delete-my-photo* option removes the file from Storage. Nothing is auto-posted
  to LinkedIn — Heal posts manually.

## 6. Decisions made where the brief left room
- **Admin role** is a `role` column on `profiles` (`student` | `admin`), checked
  via a `SECURITY DEFINER is_admin()` helper (avoids RLS recursion). A trigger
  blocks non-admins from escalating their own role.
- **Engagement dwell default** is `min_seconds: 20` (configurable per module).
- **Reported pass default** in the example is `pass_score: 70` (configurable).
- **Module id injection** is done by the `/embed` wrapper (the parent's server
  code) rather than writing into the iframe window, because the sandbox is
  opaque-origin for security. This still satisfies "the parent injects the id".
